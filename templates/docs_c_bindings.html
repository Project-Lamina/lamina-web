C bindings roadmap

The C bindings will let frontends and embedders use Lamina without depending on Rust layouts or the Rust ABI.

Status: planned

The stable lamina.h header is not shipped yet. This page records the contract we plan to implement.

Design principles

Opaque handlesC callers work with handles. Rust keeps ownership of builders, modules, JIT contexts, and diagnostics.

Explicit lifetimeEvery owned handle has a matching destroy function.

Panic containmentNo Rust panic crosses an FFI boundary.

Stable errorsCalls return status codes; copied thread-local diagnostics explain failures.

Why the builder needs an owned wrapper

The Rust IRBuilder<'a> stores borrowed names and string values. A C caller may pass a temporary char *, so the wrapper cannot retain that pointer. The C layer will copy incoming strings into an OwnedIRBuilder, then finalize an owned module before compilation.

lamina_builder_t *builder = lamina_builder_create();
lamina_builder_function(builder, "main", NULL, 0, lamina_type_i64());
lamina_builder_binary_i64(builder, LAMINA_BINARY_ADD, "answer", 40, 2);
lamina_builder_return_var(builder, lamina_type_i64(), "answer");

lamina_module_t *module = NULL;
lamina_status_t status = lamina_builder_finish(builder, &module);
lamina_builder_free(builder);

Core header shape

The public header will expose opaque handles and stable C data. It will not expose Rust enums, slices, vectors, strings, or struct layouts.

typedef struct lamina_builder lamina_builder_t;
typedef struct lamina_module lamina_module_t;
typedef struct lamina_jit lamina_jit_t;

typedef enum {
    LAMINA_OK = 0,
    LAMINA_ERROR_INVALID_ARGUMENT,
    LAMINA_ERROR_PARSE,
    LAMINA_ERROR_VALIDATION,
    LAMINA_ERROR_CODEGEN,
    LAMINA_ERROR_JIT,
    LAMINA_ERROR_IO,
    LAMINA_ERROR_INTERNAL
} lamina_status_t;

JIT lifetime rule

A lamina_jit_t handle owns executable memory. A pointer returned by lamina_jit_get_function() stays valid only while that handle is alive. The first release will also include typed i64 call helpers so common use cases do not require casting raw pointers.

Delivery checklist

  • HeaderGenerate and review a stable lamina.h.
  • ExamplesShip textual AOT, builder AOT, and JIT examples in C.
  • TestsCompile C smoke tests in CI and exercise ownership teardown.
  • DocsDocument nullability, ownership, thread behavior, and diagnostic lifetime.

Implementation order

  1. Owned IRAdd owned values, types, modules, and an owned builder layer in lamina-ir.
  2. ABI crateAdd lamina-c, opaque handles, status codes, diagnostics, and panic containment.
  3. AOTExpose textual and builder-based assembly compilation with explicit output buffers.
  4. JITWrap runtime compilation, executable-memory ownership, symbol lookup, and typed call helpers.
  5. ReleaseGenerate lamina.h, add C smoke tests, and publish examples.
